P3
Problème → Solution

Des milliers de pré-inscriptions EventMaker qui ne correspondent à aucun profil app

Le problème

Les étudiants s'inscrivent sur letudiant.fr pour le salon via EventMaker — mais n'activent jamais l'app. Leur inscription EventMaker est orpheline : elle existe dans le système de billetterie mais n'a aucun profil app associé.

Profils comportementaux impossibles à construire
Intent score ne peut pas être calculé
Exposants ne reçoivent aucune donnée sur ces visiteurs
Volume estimé : 30–40% des pré-inscrits EventMaker ne créent jamais de compte app
Solution — Webhook EventMaker

Flux de réconciliation

1
POST /api/webhooks/eventmaker reçoit l'événement
2
Extraction email → lookup table users
✓
Match exact : lier uid existant au profil EventMaker
!
Pas de match : créer profil provisoire + flag needs_review
// app/api/webhooks/eventmaker/route.ts
const { email, name, fair_id } = await req.json()
const { data: existing } = await supabase
  .from('users').select('id')
  .eq('email', email).single()

if (existing) {
  // Link uid — exact match
  await linkEventMakerProfile(existing.id, fair_id)
} else {
  // Create provisional user
  await createProvisionalUser({ email, name, fair_id })
}

P4 Problème → Solution

60%+ de no-shows — les écoles perdent des créneaux pour des fantômes

Le problème

L'étudiant réserve un créneau depuis chez lui, des semaines avant le salon. Le jour J, il ne vient pas — ou vient mais oublie son RDV. L'école perd ce créneau, qui ne peut pas être libéré pour un visiteur réel présent sur place.

✕Aucune vérification de présence physique
✕Créneaux bloqués par des fantômes distants
✕NPS exposants dégradé par les no-shows
Solution — Entry scan gate
1

Onglet RDV verrouillé

Message : "Scannez votre QR à l'entrée du salon pour accéder aux RDV"

2

Scan entrée → gate levé

entryScan enregistré → accès RDV déverrouillé instantanément

3

Étudiant réserve sur place

Réservation = présence prouvée — 0% no-shows structurels

-- Vérification gate RDV
SELECT id FROM scans
WHERE user_id = X
  AND channel = 'entry'
  AND created_at >= today
-- Si 0 résultat → RDV bloqué
Impact : 0% no-shows structurels · NPS exposants ↑
P5
Problème → Solution

On ne sait pas combien de temps chaque visiteur passe dans le salon

Parcours capturé — channels de scan
IN — Entrée salon
channel: entry · timestamp T₀
Stands × N
channel: stand · timestamps T₁…Tₙ
OUT — Sortie salon
channel: exit · timestamp Tₓ
Calcul & stockage

Formule dwell time

dwell_minutes =
(exit.created_at − entry.created_at)
/ 60 000

Stocké sur users.last_dwell_minutes après chaque salon

Intent score enrichi

>90 min → intent score +15 pts
Corrélation dwell ↔ intent dans le rapport J+1

Cas d'usage data

Heatmap fréquentation par heure (peak hours)
Corrélation dwell ↔ intent score
Enrichissement rapport J+1 exposants

Personas

4 profils de visiteurs sans app — une stratégie pour chacun

Chaque visiteur non-app a un parcours de capture défini dans le système

👻
P1 — Ghost pré-inscrit

Inscrit sur letudiant.fr pour le salon mais jamais activé l'app. Profil EventMaker orphelin.

Friction :

Profil EventMaker orphelin sans uid app

→ Webhook EventMaker — réconciliation automatique
🏫
P2 — Élève de groupe

Vient avec sa classe. Le prof a inscrit le groupe mais pas les élèves individuellement.

Friction :

30 élèves comptés comme "1 groupe" = 0 profil individuel

→ Lien prof avant salon / Score Booth fallback
🚶
P3 — Walk-in spontané

Passe devant le salon, entre spontanément. Aucune préparation, aucune inscription préalable.

Friction :

Représente ~60% du trafic, 100% anonyme sans Score Booth

→ Score Booth = solution principale
👪
P4 — Accompagnant adulte

Parent qui accompagne son enfant. Non-étudiant, présent physiquement mais sans rôle app.

Friction :

Aucun rôle dans l'app actuelle — présence non documentée

→ Module parent read-only (sprint 3)
Stack technique

Stack & choix structurants

Chaque choix tech est motivé par une contrainte produit spécifique

⚡

Next.js 16.2.2

App Router, Server Components, SSR. API routes dans /app/api. Layout par rôle isolé.

🗄️

Supabase

SSR client + adminClient (service role). RLS par rôle. Realtime pour dashboard admin.

📱

PWA offline-first

Manifest + service worker. Score Booth fonctionne sans connexion. QR cachés localement.

🔒

RGPD / RLS

Row Level Security Supabase. Exposant = agrégé uniquement. Profils = propriété L'Étudiant.

🎨

Design System Saguez

Composants UI L'Étudiant. Tokens couleur. Typography DM Sans. Cohérence marque.

🚀

Vercel Deploy

Preview sur PR. Production sur main. Variables env Supabase injectées. Edge functions.
