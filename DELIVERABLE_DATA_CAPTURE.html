Section 00

Le problème : 60 à 70 % des visiteurs sont invisibles

Aujourd'hui, L'Étudiant ne collecte des données que sur les étudiants ayant téléchargé l'application. Tous les autres visiteurs du salon quittent le Palais des Congrès sans laisser de trace — ni pour L'Étudiant, ni pour les établissements exposants.

📐

Hypothèses de simulation. Les chiffres ci-dessus sont des estimations construites à partir de benchmarks de collecte sur des salons professionnels B2C (Educations, SIAL, Mondial de l'Auto). Aucune donnée réelle L'Étudiant n'est disponible à ce stade. Chaque hypothèse doit être validée ou corrigée par l'équipe produit lors de la session du 19 avril.

🎯

Opportunité concrète. Un étudiant sans app n'est pas non-traçable — il est simplement non-incité. Les études de comportement mobile montrent que 73% des 16–22 ans scannent un QR code si le bénéfice immédiat est clair et que la friction reste sous 45 secondes. C'est la fenêtre que cette architecture exploite.

⚖️

Contrainte RGPD transversale — Mineurs. Le Salon de l'Étudiant reçoit un public majoritairement composé de lycéens de 15 à 18 ans. L'article 8 du RGPD et la loi Informatique et Libertés imposent le consentement parental pour tout traitement marketing des moins de 15 ans. Toute interface de collecte doit poser une vérification d'âge avant toute capture de PII. Un traitement spécifique “junior” est défini en Section 04.

Section 02

Stack d'incitation — Ce que chaque tier de données débloque

L'architecture repose sur un principe d'échange de valeur explicite et progressif : plus l'étudiant partage, plus il reçoit. Chaque tier doit avoir une valeur perçue immédiatement supérieure à l'effort demandé.

Tier 0 · Anonyme
Données demandées : Aucune
Reçoit : Vue standard du salon (carte générique)
Tier 1 · Profil minimal
Données : Niveau, Domaine, Région
Effort : 3 clics — 30 secondes
🗺️ Carte du salon personnalisée avec les stands correspondant à son profil, surlignés en rouge
Tier 2 · Email + intérêt implicite
Données : Email + prénom + stands scannés
Effort : 1 champ email par stand visité — 20 secondes
📦 Brochures de toutes les écoles scannées dans un email unique · Pas de sac papier
Tier 3 · Profil complet
Données : Tier 2 + Bac série, filières déclarées, alternance, région
Effort : 5 questions — 90 secondes (borne ou QR)
📊 Score d'orientation 0–100 · Top 5 établissements personnalisés avec explication des matchs
Tier 4 · Optin WAX + compte créé
Données : Tier 3 + consentement marketing WAX
Effort : Créer un compte app ou web · 2 minutes
🏆 Bilan salon complet J+1 · Recommandations personnalisées · Alertes JPO · Accès au comparateur d'écoles full
Tier Données L'Étudiant récupère Valeur pour les exposants Benchmark conversion
Tier 0 Session anonyme, timestamp entrée Comptage visiteurs (déjà fait) 100% des visiteurs
Tier 1 Profil déclaratif minimal, session liée Audience analytics par filière/région 40–55% (plan = utilité directe)

🔑

Règle de déduplication. L'email est la clé primaire d'identité. Si un visiteur scanne 3 stands différents avec 3 emails différents (peu probable), chaque adresse crée un guest_lead distinct. Si la même adresse revient sur plusieurs stands ou mécanismes, tous les scans sont consolidés sous un seul profil. La déduplication se fait en temps réel via une lookup sur la table guest_leads(email) avant tout INSERT.

Session cookie. Le session_token (cookie navigateur, 30 jours, same-site=strict) lie les scans effectués depuis le même appareil avant et après la capture d'email. Si l'étudiant change d'appareil, les deux sessions restent distinctes — pas de tracking cross-device sans consentement.

Section 04

Matrice RGPD — Base légale, consentement et rétention par point de collecte

Chaque donnée collectée doit être justifiée par une base légale Art. 6 RGPD avant d'être implémentée. Aucun champ n'entre en base sans que cette colonne soit remplie et validée DPO.

⚠ Traitement spécial — Mineurs de moins de 15 ans
  • RGPD Art. 8 + Loi Informatique et Libertés : le consentement d'un mineur de moins de 15 ans (France) est invalide sans autorisation parentale pour tout traitement marketing.
  • Chaque formulaire de collecte (M1, M2, M3, M4) doit afficher la question "Êtes-vous âgé de 15 ans ou plus ?" avant tout champ PII.
  • Réponse NON → collecte de PII bloquée. Seule la délivrance du service (brochure, slides) est autorisée via email, sans stockage long terme ni optin marketing.
  • Réponse OUI → flux normal. Pas de vérification documentaire requise (déclaration suffisante, responsabilité transférée à l'utilisateur).

Section 05

Extension du schéma de base de données

3 nouvelles tables pour les visiteurs hors-app. Elles s'intègrent au schéma existant (Supabase / PostgreSQL) et permettent la résolution d'identité vers la table users existante.

-- ═══════════════════════════════════════════════════════════════════════════
-- Migration 003 — Collecte hors-app : visiteurs sans compte
-- Run AFTER 002_appointments_consent.sql
-- ═══════════════════════════════════════════════════════════════════════════

-- ─── SESSIONS ANONYMES ────────────────────────────────────────────────────
-- Créée dès le premier scan QR, même sans email.
-- session_token = UUID généré côté client (localStorage + cookie httpOnly)
create table if not exists public.guest_sessions (
  id               uuid primary key default uuid_generate_v4(),
  session_token    text not null unique,        -- UUID stocké en cookie navigateur
  event_id         uuid not null references public.events(id),
  is_minor         boolean default false,         -- déclaré via question d'âge
  declared_level   text,                           -- Terminale / BTS / Licence…
  declared_domain  text,                           -- domaine d'intérêt M1
  declared_region  text,                           -- département / région M1
  device_hint      text,                           -- hash du user-agent (non PII)
  first_seen_at    timestamptz default now(),
  last_seen_at     timestamptz default now(),
  resolved_user_id uuid references public.users(id)  -- rempli si identité résolue
);

-- ─── SCANS DES VISITEURS HORS-APP ─────────────────────────────────────────
-- Chaque stand scanné par un visiteur sans app crée une ligne ici.
create table if not exists public.guest_scans (
  id               uuid primary key default uuid_generate_v4(),
  session_id       uuid not null references public.guest_sessions(id) on delete cascade,
  event_id         uuid not null references public.events(id),
  school_id        uuid references public.schools(id),  -- null si conf générale
  session_conf_id  uuid references public.sessions(id), -- null si stand
  mechanism        text check (mechanism in ('map_qr', 'brochure_qr', 'conf_checkin', 'kiosk')),
  incentive_served text,                           -- ex: 'brochure_ensad_2026.pdf'
  email_captured   boolean default false,
  scanned_at       timestamptz default now()
);

-- ─── LEADS HORS-APP IDENTIFIÉS (email collecté) ───────────────────────────
-- Créé dès qu'un email est fourni via M2, M3 ou M4.
-- Peut coexister avec un users.id si l'email matche un compte existant.
create table if not exists public.guest_leads (
  id                   uuid primary key default uuid_generate_v4(),
  session_id           uuid references public.guest_sessions(id),
  email                text not null,
  first_name           text,
  is_minor             boolean default false,
  optin_letudiant      boolean default false,    -- marketing L'Étudiant
  optin_wax            boolean default false,    -- recommandations WAX
  consent_ip_hash      text,                       -- SHA-256 de l'IP (non PII)
  consent_date         timestamptz default now(),
  consent_version      text not null default '1.0',
  declared_level       text,
  declared_bac_series  text,
  declared_domains     text[] default '{}',
  declared_region      text,
  wants_alternance     boolean,
  orientation_score    integer,                   -- calculé en J+1 post-salon
  recap_email_sent_at  timestamptz,
  resolved_user_id     uuid references public.users(id), -- si conversion vers compte
  created_at           timestamptz default now(),
  unique(email, session_id)
);

-- ─── INDEX ────────────────────────────────────────────────────────────────
create index if not exists idx_guest_sessions_token   on public.guest_sessions(session_token);
create index if not exists idx_guest_sessions_event   on public.guest_sessions(event_id);
create index if not exists idx_guest_scans_session    on public.guest_scans(session_id);
create index if not exists idx_guest_scans_school     on public.guest_scans(school_id);
create index if not exists idx_guest_leads_email      on public.guest_leads(email);
create index if not exists idx_guest_leads_resolved   on public.guest_leads(resolved_user_id) where resolved_user_id is not null;

-- ─── RLS ──────────────────────────────────────────────────────────────────
-- Tables publiques en INSERT (non authentifié = scan QR)
-- SELECT réservé service_role (admin / backend)
alter table public.guest_sessions enable row level security;
alter table public.guest_scans enable row level security;
alter table public.guest_leads enable row level security;

create policy "Anyone can create a guest session"
  on public.guest_sessions for insert with check (true);

create policy "Anyone can record a guest scan"
  on public.guest_scans for insert with check (true);

create policy "Anyone can create a guest lead"
  on public.guest_leads for insert with check (true);
🔄

Job de résolution d'identité (Edge Function Supabase — J+1, 6h00). La nuit suivant le salon, une Edge Function effectue trois opérations : (1) pour chaque guest_lead, vérifier si l'email existe dans users — si oui, créer les scans et leads correspondants sur le compte existant et marquer resolved_user_id ; (2) calculer le orientation_score à partir des guest_scans et du profil déclaré ; (3) déclencher l'envoi de l'email de récap M5 pour tous les guest_leads sans compte app. Les données anonymes non résolues sont conservées 6 mois puis purgées par un CRON job.
